BLOCKTOWN

在不規則格網上，蓋一座奶油色小鎮

TL;DR

用瀏覽器即時建造的 3D 小鎮玩具，模仿 Townscaper 風格。屋頂、門窗、欄杆、簷口全部自動生成。純原生 HTML/CSS/JS、沒 build 工具，打開網頁就跑、60fps。

Blocktown 是一個瀏覽器 3D 建築玩具，受 Oskar Stålberg 的 Townscaper 啟發。透過程序生成不規則格網、Loop subdivision 圓潤建築、地基紅色 cornice 與鑄鐵欄杆、3+3 種凹陷感門窗變體、動態水波紋 + foam，整體在 60fps 即時運算。

核心特色

01 · TERRAIN

程序生成地形

第一步：起點。從整齊的方格網開始，用一個圓形 mask 切出小島輪廓——所有不在圓內的格子都丟掉。

第二步：變有機。對整片格網施加波形扭曲，讓行與列像水波一樣彎，產生 Townscaper 風的「流動感」；再對內部頂點加上微抖動，打破過於規則的感覺。

第三步：收斂。跑等邊鬆弛把所有 quad 的邊長磨到差不多；順帶過濾掉因為 mask 切出來的離島碎塊。整段過程邊界頂點全程鎖定不動，確保外輪廓始終是圓島。最終產出 441 個 quad cells。

Why：單純方格太呆板、純隨機會崩成碎片。「先扭曲、再抖動、再等邊化」三步漸進，每一步都退一點規則、多一點有機，是兼顧「不亂」與「不呆」的中間解。

02 · VISUAL

圓潤 + 桶身 + 程序材質

第一步：揉圓。用 Loop subdivision 把硬邊的方塊揉成 puffy 圓潤形狀，像揉麵團一樣把所有銳角都磨成弧線。每個方塊的顏色直接「烘」進頂點，揉圓時也會自然平滑過渡。

第二步：桶身。地基底部往外擴一點，形成 Townscaper 招牌的「碗形地基浮在水面」造型。樓層上的地板獨立一層 mesh，讓簷口能緊貼邊緣。

第三步：材質。屋頂自動鋪上瓦片圓點、牆面加上水平木紋；水面跑動態波紋與泡沫；天空有自然飄移的雲層；最後整張畫面套上電影感色調與柔和暈光。

Why：方塊原本是硬邊立方體，跟 Townscaper 那種「碗形地基浮在水面」的視覺差很遠。揉圓 + 桶身 + 程序材質三層疊，才生出這種「夢幻又溫暖」的整體氛圍。

03 · AUTO-DECO

自動裝飾系統

第一步：屋頂自動長。程式判斷每塊方塊周圍格子的狀況，自動決定屋頂該長成金字塔、屋脊還是平頂；屋脊兩端會自動升起垂直三角牆。

第二步：屋簷自動推。屋頂外緣統一向外突出形成屋簷；遇到 L 形凹角時自動收進來，形成圓滑的過渡曲線，避免穿模或裂縫。

第三步：配件自動裝。地基頂自動包上紅色簷口帶與鑄鐵欄杆；每棟建築外圍再套上一圈水面泡沫光暈。

第四步：門窗自動選。三種門 + 三種窗的變體，每棟建築用自己的編號隨機選一組樣式；外框前置 + 玻璃凹進去一層，從外看像「往牆內看進去」的洞，門窗看起來有深度。

Why：讓玩家為每棟建築決定 30 個細節（屋頂形狀、欄杆、門窗位置、凹角怎麼處理 …）會崩潰。Townscaper 最神奇的就是「程式幫你想好所有細節」，本作從屋頂到門窗全部自動生成，玩家只負責放方塊。

Curve Distortion · 雙層 sine 波

const phaseX = rnd() * Math.PI * 2;
const phaseZ = rnd() * Math.PI * 2;
for (const v of usedVerts) {
  const [x, z] = pts[v];
  // x 位移取決於 z（垂直波）
  const dx = Math.sin(z * CURVE_FREQ + phaseX) * CURVE_AMP
           + Math.sin(z * CURVE_FREQ * 1.73 + phaseX + 1.1) * CURVE_AMP * 0.4;
  // z 位移取決於 x（水平波）
  const dz = Math.cos(x * CURVE_FREQ + phaseZ) * CURVE_AMP

}

Why：雙層相位錯開的 sine 波（振幅 0.35、頻率 0.32），讓行列像水波一樣彎，產生 Townscaper 風 flow 感；seed 影響相位 → 每次 shuffle 形狀不同。

for (let pass = 0; pass < SHAPE_PASSES; pass++) {
  let totalLen = 0;
  for (const [a, b] of edgeList) totalLen += Math.hypot(...);
  const targetLen = totalLen / edgeList.length;  // 目標 = 平均邊長

  for (const [a, b] of edgeList) {

    const half = (len - targetLen) * 0.5;  // 太長→端點互推；太短→端點互拉
    disp[a][0] += (dx/len)*half;  disp[b][0] -= (dx/len)*half;

  }
  for (let i = 0; i < pts.length; i++) {
    if (isBoundaryVert.has(i)) continue;   // 邊界鎖定 → 圓島輪廓不變
    pts[i][0] += disp[i][0] / cnt[i] * SHAPE_STRENGTH;
  }
}

Why：10 pass 漸進把所有 quad 的邊長磨到差不多；邊界頂點全程鎖定，外輪廓始終是圓島。比 Laplacian smoothing 多保留了 curve distortion 的彎曲感。

keepLargestComponent · 離島過濾

main.js:590 ↗ 
const visited = new Uint8Array(allCells.length);
const components = [];
for (let i = 0; i < allCells.length; i++) {
  if (visited[i]) continue;
  const comp = [];
  const stack = [i];           // BFS / DFS 走訪整塊連通分量
  visited[i] = 1;
  while (stack.length) {
    const cid = stack.pop();
    comp.push(cid);
    for (const nb of allCells[cid].neighbors) {
      if (nb !== null && !visited[nb]) {
        visited[nb] = 1;
        stack.push(nb);
      }
    }
  }
  components.push(comp);
}
components.sort((a, b) => b.length - a.length);
return new Set(components[0]);   // 只保留最大的那塊

Why：圓形 mask 切完可能產生孤立的小碎塊；BFS 走訪 + 取最大連通分量 → 保證玩家拿到的永遠是「一整座島」而不是散落的碎片。

// 4 階 FBM：自然雲團
float fbm(vec2 p) {
  float v = 0.0, amp = 0.5;
  for (int i = 0; i < 4; i++) {
    v += amp * vnoise(p);
    p *= 2.07;  amp *= 0.55;
  }
  return v;
}
void main() {
  float h = vDir.y;
  // 高度漸層：地平線只佔窄一段、主要是藍天
  vec3 col = (h >= 0.0)
    ? mix(uHorizon, uZenith, smoothstep(0.0, 0.18, h))
    : mix(uHorizon, uGround, smoothstep(0.0, -0.15, h));

  // 雲層：球面 lat/lon 當 noise 輸入
  float az = atan(vDir.x, vDir.z), el = vDir.y;
  vec2 cp = vec2(az * 1.4, el * 3.0);
  cp.x += uTime * 0.04;                  // 雲團繞天飄移
  float cloud = smoothstep(0.50, 0.62, fbm(cp * 1.2));
  col = mix(col, uCloudColor, cloud * 0.95);
  gl_FragColor = vec4(col, 1.0);
}

Why：4 階 FBM 比單層 noise 自然 5 倍；窄閾值（0.50→0.62）讓雲跟藍天邊界分明、不糊；uTime * 0.04 是雲團漂移速度，數值很小但日積月累就動了。

}

Why：每棟建築一圈 foam ring（Shape + hole 環形 mesh）；自訂 radialT attribute 讓 fragment shader 知道每個頂點離建築多遠，uTime 驅動同心波 → 視覺效果像水拍上岸。

function handleClick(ev, isRight) {
  const hit = pick(ev.clientX, ev.clientY);   // Raycaster 命中
  if (!hit) return;
  const ud = hit.object.userData;

  if (isRight) {                              // 右鍵移除頂層
    if (ud.type === 'column' || ud.type === 'foundation_top') {
      removeTopBlock(closestCellAt(hit.point.x, hit.point.z));
    }
    return;
  }
  if (ud.type === 'ground') return addBlock(ud.cellId, 0);   // 空地 → 地基
  if (ud.type === 'foundation_top') {
    return addBlock(cellId, cells[cellId].blocks.length);    // 地基頂 → 疊樓
  }
  if (ud.type === 'column') {
    const wallTop = cellTopY(cells[cellId]);
    // 屋頂三角化讓 normal 不可靠 → 改用 hit.point.y 判定
    if (hit.point.y > wallTop - 0.05) {
      addBlock(cellId, cells[cellId].blocks.length);         // 屋頂區域 → 疊
    } else {
      // 牆面 → 朝 normal 外側推 0.6 找鄰格
      const target = closestCellAt(p.x + n.x*0.6, p.z + n.z*0.6);
      addBlock(target, cells[target].blocks.length);
    }
  }
}

Why：屋頂三角化導致法線方向混亂（朝下底面、水平側壁都摻在一起），純看 n.y 會誤判。改用 hit.point.y > wallTop 判定「屋頂區域」，誤判率歸零。

  • 3 種門 + 3 種窗 variant：拱形 / 矩形雙開 / 單門+雨遮 · 拱窗 / 4-pane 十字框 / 圓窗；每棟用 bid hash 固定一種。

挑戰與解決

CASE 01

兩階段建造的設計演進

問題：最初所有格子第一次點擊都當「樓層」，但矮層樓在水面平行視角看起來像漂浮的方塊，沒有「島」的厚度。

解：抽出第一層為「地基」podium：從水下 WATER_Y − 0.20 延伸到 FOUNDATION_TOP_Y = 0.3，第二次起才是樓層 + 屋頂。地基頂還加紅色 cornice 帶與鑄鐵欄杆強化 Townscaper 視覺。

CASE 02

rebuildAll 順序 bug

問題：原本 rebuildAll 先用 buildingCells 重建 mesh，再 refreshDecorations 更新 buildingCells。第一次種 demo town 時跑進空陣列，沒生出任何 mesh。

解：把 refreshDecorations() 移到 mesh rebuild 之前，先重算分組再用新的 buildingCells 迭代。

CASE 03

屋簷穿模

問題：屋簷外推會造成多種破圖：鄰格較高時插進牆裡；多 cell 排列時邊緣不均勻；ridge 端點縮回成裂縫；gable 牆頂與屋頂 apex 共平面 Z-fight 破圖。

解：四刀並行 — (a) 跨 region 法線累加：所有 boundary vert 統一外推 22cm，連續均勻不縮回；(b) 垂直 gable 牆：牆頂 Y 跟 roofBase 升到 ridge 高度形成三角牆；(c) Z-fight cap：牆頂 Y 限制 apex − 10cm；(d) 點擊判定用 hit 點 Y 而非 normal.y。

CASE 04

稀疏陣列導致 undefined.color

問題：seedDemoTown 用條件機率「跳過某層」造成 blocks 陣列稀疏（blocks[1] undefined 但 blocks[2] 有值），rebuildBuildingMesh 讀到 undefined.color 炸掉。

解：改用 blocks.push(...) 強制連續疊，邏輯改成 nested if 確保上層只在下層存在時才疊。

CASE 05

選單繞鏡頭的「停頓感」

問題：menuOrbitT 寫死 (1/60) 累加 + shadow.autoUpdate=true + 4096² 陰影貼圖，遇 GC 或 frame drop 時鏡頭看起來會停住一拍。

解：三刀並行 — (a) dt-based 累加且 clamp 100ms 上限；(b) shadow.autoUpdate=false 只在場景變動才 flag；(c) mapSize 4096²→2048²。穩定 60fps。

一路上學到什麼

💡 效能最佳化不是空想出來

原本以為「shader 應該很慢」拍腦袋猜原因，profile 後才發現是 4096² shadow map 每幀重算，跟 shader 沒關係。先量測再下刀，比猜十次都準。

🐛 大多數 bug 都來自順序假設

rebuildAll 順序錯、seedDemoTown 稀疏陣列、屋簷外推方向，都是「我以為 A 會比 B 先發生」這類隱性假設。把順序寫成註解是最便宜的防線。

🎨 UX 細節比功能更耗時

Hover ghost、平滑追焦鏡頭、cinematic 選單、鍵盤導航、PROJECT INFO 這頁本身花的時間 是核心建造邏輯的 2 倍。但這些才是讓人記得住的部分。
